# 1:1 Review — CarroQueSí — 2026-07-22

## Overview
Seventh review, and the first window in five that was a genuine feature build rather than test infra, refactors, and bug fixes: Siri Shortcuts integration (#111) with a new auth strategy and ADR-006, a Siri naming fix (#112), and #113 (JAV-43), which replaced part of what #111 had shipped a day earlier. Two ADRs landed unprompted — a first. Main theme: the design-stage failure space is under-explored. #111's `list_id="default"` resolver ("most-recently-updated list") was non-deterministic in a shared-list app, and that was derivable from the data model alone — no device, no user needed. The user caught it by thinking it through, not me.

## AI: Self-Assessment

**Wins**
- ADR-006 (API key auth for non-browser clients) and ADR-007 (per-user default list) were both written proactively, without being asked — the first time the ADR habit has fired on its own rather than as an assigned item
- #113's design is genuinely careful: the `updated_at` ordering heuristic lives **only** in the migration backfill, never in the request path, so existing users keep their prior target without leaking non-determinism into the runtime; and `PUT /lists/{id}/default` deliberately doesn't bump `lists.updated_at`, correctly treating `is_default` as per-member state that shouldn't trigger a re-fetch for co-members on the 5s poll
- Resolved the carried-over Docker/ADR question in-session rather than filing it a third time

**Areas for Improvement**
- **The core miss: I explored the solution space in #111's `/brainstorming`, not the failure space.** ADR-006 does a real job on auth tradeoffs (static key vs. OAuth vs. Firebase custom token). But the resolver semantics sitting right next to it got no equivalent scrutiny. "Most-recently-updated list" breaks the moment a second person edits a shared list — Siri lands on whichever list your partner last touched. That's a UX failure wearing a technical costume, and it's exactly the class tests never catch.
- #113 was, functionally, the user doing the design review I should have done a day earlier.
- Pushback at the design stage has been too soft — presenting considerations rather than taking positions. Confirmed directly by the user when asked.
- Carried the Docker/ADR item for a week without acting despite reading AGENTS.md every session. The two ADRs I *did* write were attached to a PR I was actively building; the assigned one had no PR to hang on. (User's calibration: this was mostly a consequence of sessions going straight to specific issues — not a standalone failure. Noted, but the mechanism gap is real.)

**Commitments**
- In every `/brainstorming` session, add an explicit **failure-space section** before converging: how does this break with multiple users, with state changing between calls, with zero items, with many. Named as its own section, not woven into prose.
- **Take positions, not postures.** When the preferred option looks wrong, say "this is wrong because X" — not "one consideration might be."
- Front-load UX forks especially. Objective bugs get caught by tests; subjective/UX ones need a human in the loop *before* attention is committed, which means surfacing the forks rather than quietly picking one.
- Carried: verify environment-conditional logic explicitly (still no clean test case — third review).

## Your Workflow: Observations & Recommendations

**What's Working**
- Catching the default-list UX problem by thinking it through post-ship and immediately ideating a cheap alternative — the fix landed within a day, before any user could depend on the old behaviour
- Clean separation of *professional capability* goals from *product* bets when pressed on the parked items — loops and MCP had been wrongly bucketed together as "parked research"; they're not the same kind of thing, and only the second is subject to a feature-creep test
- Right-sizing to project stage remains consistent: release cadence explicitly declined as a non-problem while the user is effectively the only user

**Opportunities**
- **Drop MCP from the tracked list.** It fails the stated bar — "regular users integrating the app into their daily workflows." MCP clients are Claude Desktop, Cursor, and similar; that's a developer audience. Building it would be building for yourself while calling it a user feature. Revisit only if that audience assumption changes.
- **Siri stays, but for a stated reason: the writeup.** Pushing it further is justified by the blog-post/insight goal even if usage never exceeds one person. That's a legitimate rationale that "does anyone use this" can't argue against — worth keeping explicit so it doesn't drift into unexamined investment.
- **The friction signal is worth acting on while it's fresh.** Describing a three-day-old feature as "cumbersome to set up and clunky to use" is the most useful product data in this window. #113 already improved the setup path; the remaining friction is the material the blog post is actually about.

**Resolved this session**
- Docker-isolated baseline generation does **not** warrant an ADR. `frontend/tests/README.md` already documents every tradeoff (Linux-not-macOS, amd64 pinning, the two named volumes and why each exists). ADRs here record architectural choices with long-lived runtime consequences; this is reversible build tooling whose rationale is better colocated with the thing it explains. Item closed.
- Release cadence: explicitly a non-issue at current scale. "Cut when it feels right" is the standing answer — closed, not carried.
